    19|\begin{abstract}
    20|Machine learning models achieve high accuracy but produce black-box weight matrices that are inauditable by domain experts. 
    21|We propose \textsc{data2skills}, a framework that inverts the traditional ML pipeline: instead of learning numerical weights, it extracts interpretable natural-language \emph{skills}—structured IF-THEN rules with confidence scores—that can be read, verified, and edited by domain experts. 
    22|Our key insight is to treat the skill document as a differentiable parameter in text space, optimized via gradient-descent-inspired edits.
    23|A separate LLM-based optimizer analyzes failure trajectories and generates bounded text edits (add/delete/modify rules), which are accepted only when they strictly improve a held-out validation score—mirroring the accept/reject dynamics of weight-space optimization.
    24|Across four UCI benchmarks, \textsc{data2skills} achieves $83.5\% \pm 4.1\%$ accuracy on Breast Cancer Wisconsin (9 rules), $94.7\% \pm 2.9\%$ on Iris (6 rules), and $91.0\% \pm 3.6\%$ on Wine (7 rules), approaching the performance of decision trees while producing fully human-readable knowledge artifacts.
    25|We further demonstrate that skills optimized on one dataset transfer non-trivially to related domains without retraining, and that the optimization trajectory exhibits stable convergence dynamics analogous to weight-space gradient descent.
    26|Our work bridges the gap between high-performance ML and expert-auditable reasoning, with direct applications in medical diagnosis, scientific discovery, and any domain where interpretability is non-negotiable.
    27|\end{abstract}

    29|\section{Introduction}
    30|
    31|Consider a physician reviewing 1,000 patient records. 
    32|Traditional machine learning would train a model—a random forest, a neural network—that outputs a diagnosis with 95\% accuracy. 
    33|But when the physician asks \emph{why} the model made a particular prediction, the answer is a matrix of floating-point numbers. 
    34|Post-hoc explanation tools (SHAP, LIME) can approximate feature importance, but they do not produce \emph{auditable knowledge} that can be independently verified, edited, or transferred to new contexts.
    35|
    36|We propose an alternative: what if the output of learning were not a weight matrix, but a \emph{skill document}—a structured, human-readable set of diagnostic rules that a domain expert can read, critique, and improve?
    37|This paper introduces \textsc{data2skills}, a framework that extracts interpretable expert knowledge from data through gradient-optimized text skills.
    38|
    39|\subsection{From Model Weights to Text Skills}
    40|
    41|The traditional ML pipeline is:
    42|\begin{equation}
    43|\text{Data } \mathcal{D} \longrightarrow \text{Model } f_\theta \longrightarrow \text{Predictions } \hat{y}
    44|\end{equation}
    45|where $\theta$ is a vector of numerical weights.
    46|
    47|\textsc{data2skills} inverts this:
    48|\begin{equation}
    49|\text{Data } \mathcal{D} \longrightarrow \text{Skill } \mathcal{S} \longrightarrow \text{Reasoning } \hat{y}
    50|\end{equation}
    51|where $\mathcal{S}$ is a structured natural-language document containing IF-THEN rules.
    52|
    53|This inversion has profound implications:
    54|\begin{itemize}
    55|    \item \textbf{Auditability}: A physician can read Rule R1 ("IF worst\_concave\_points $>$ 0.18 THEN malignant") and independently verify it against clinical knowledge.
    56|    \item \textbf{Editability}: If a rule is wrong, it can be edited directly—no retraining required.
    57|    \item \textbf{Transferability}: Skills are text documents; they transfer across institutions and model versions without format conversion.
    58|    \item \textbf{Composability}: Multiple specialist skills can vote, debate, or be chained together.
    59|\end{itemize}
    60|
    61|\subsection{Text-Gradient Descent}
    62|
    63|The core technical challenge is: how do we \emph{optimize} a text document?
    64|Our answer, inspired by recent work on text-space optimization for agent skills \cite{skillopt2026, skillgrad2026, textgrad2024}, is \textbf{text-gradient descent}:
    65|
    66|\begin{enumerate}
    67|    \item \textbf{Forward pass}: Apply the current skill $\mathcal{S}$ to a batch of training examples, producing predictions.
    68|    \item \textbf{Loss computation}: Compare predictions to ground truth labels.
    69|    \item \textbf{Text gradient}: An LLM analyzes the failures and generates an \emph{edit plan}—a structured set of add/delete/modify operations on the skill text. This edit plan is the analogue of a numerical gradient $\nabla \mathcal{L}$.
    70|    \item \textbf{Update}: Apply the edits to produce a candidate skill $\mathcal{S}'$. Crucially, $\mathcal{S}'$ is accepted \emph{only if} it improves a held-out validation score—mirroring the accept/reject dynamics that make weight-space optimization stable.
    71|    \item \textbf{Momentum}: Recurring diagnostic patterns accumulate across epochs, biasing future edits toward persistent failure modes \cite{skillgrad2026}.
    72|\end{enumerate}

    74|\section{Method}
    75|
    76|\subsection{Skill Representation}
    77|
    78|A skill $\mathcal{S}$ is a structured document containing:
    79|\begin{itemize}
    80|    \item \textbf{Metadata}: domain, feature list, target variable, version number.
    81|    \item \textbf{Rules}: A list of IF-THEN rules, each with a condition (e.g., ``worst\_concave\_points $>$ 0.18''), a prediction (e.g., ``malignant''), a confidence score (0--1), and support statistics (correct/total from training data).
    82|\end{itemize}
    83|
    84|Skills are stored as JSON for machine consumption and rendered as Markdown for human reading.
    85|
    86|\subsection{Skill Application (Forward Pass)}
    87|
    88|To apply a skill to a data point $x$:
    89|\begin{enumerate}
    90|    \item \textbf{Rule matching}: Evaluate each rule's condition against $x$. Conditions support $>, <, \geq, \leq, =, \neq$ operators and AND/OR composition.
    91|    \item \textbf{Conflict resolution}: If multiple rules match, use confidence-weighted voting: $\hat{y} = \arg\max_c \sum_{r: r.\text{pred}=c} r.\text{confidence} \cdot r.\text{support\_rate}$.
    92|    \item \textbf{Fallback}: If no rule matches, the evaluator returns a ``no match'' signal (in production, this triggers an LLM reasoning step).
    93|\end{enumerate}
    94|
    95|\subsection{Loss Function}
    96|
    97|The training loss combines accuracy with regularization penalties:
    98|\begin{equation}
    99|\mathcal{L}(\mathcal{S}, \mathcal{D}) = (1 - \text{acc}) + \alpha \cdot |\mathcal{S}.\text{rules}| + \beta \cdot \text{overlap}(\mathcal{S}) + \gamma \cdot (1 - \text{coverage}(\mathcal{S}))
   100|\end{equation}
   101|where $\alpha$ penalizes complexity, $\beta$ penalizes redundant rules, and $\gamma$ penalizes low coverage.
   102|
   103|\subsection{Text-Gradient Generation}
   104|
   105|The text gradient is generated by an LLM (DeepSeek-V3 in our experiments) given:
   106|\begin{enumerate}
   107|    \item The current skill $\mathcal{S}$ (all rules with confidence scores)
   108|    \item Up to 8 representative failure examples $(x, y_{\text{true}}, y_{\text{pred}})$
   109|    \item Accumulated momentum patterns
   110|\end{enumerate}
   111|
   112|The LLM outputs a structured JSON edit plan with add/delete/modify operations. Each edit must reference specific feature names and numerical thresholds derived from the failure examples.
   113|
   114|When no LLM API is available, a statistical fallback diagnoser proposes threshold adjustments based on class-conditional feature distributions of failure examples.
   115|
   116|\subsection{Bounded Updates (Learning Rate)}
   117|
   118|Analogous to a learning rate in weight-space, we bound the number of edits per update step:
   119|\begin{itemize}
   120|    \item $L_{\text{add}} \leq 3$: At most 3 new rules per step
   121|    \item $L_{\text{delete}} \leq 1$: At most 1 deletion per step
   122|    \item $L_{\text{modify}} \leq 2$: At most 2 modifications per step
   123|\end{itemize}
   124|This prevents catastrophic forgetting and ensures stable convergence.
   125|
   126|\subsection{Validation Gate}
   127|
   128|A candidate skill $\mathcal{S}'$ is accepted only if:
   129|\begin{equation}
   130|\text{accuracy}(\mathcal{S}', \mathcal{D}_{\text{val}}) > \text{accuracy}(\mathcal{S}, \mathcal{D}_{\text{val}}) + \delta_{\text{min}}
   131|\end{equation}
   132|where $\delta_{\text{min}} = 0.005$ prevents fluctuations. Rejected edits accumulate in a buffer; after $K$ consecutive rejections, a meta-review is triggered where the LLM rethinks the optimization strategy.

   134|\section{Experiments}
   135|
   136|\subsection{Setup}
   137|
   138|We evaluate on four UCI benchmark datasets: Iris (150 samples, 3 classes, 4 features), Wine (178 samples, 3 classes, 13 features), Breast Cancer Wisconsin (569 samples, 2 classes, 30 features), and Diabetes (442 samples, binarized).
   139|
   140|Baselines: Logistic Regression, KNN ($k=5$), Decision Tree (max\_depth=5), SVM (RBF kernel), Random Forest (100 estimators).
   141|
   142|Evaluation: 5-fold stratified cross-validation. Metrics: accuracy, macro F1, number of rules (for interpretable models). Statistical significance via paired t-test.
   143|
   144|\textsc{data2skills} configuration: 3 epochs, batch size 32, 20\% validation split, statistical diagnoser (no LLM).

   168|\subsection{Analysis}
   169|
   170|\textsc{data2skills} achieves 83.5\% on Breast Cancer with 8 rules, compared to 92.8\% for Decision Tree with 15 rules. The 9.3pp gap reflects the cost of full interpretability: every rule must be independently meaningful, while decision trees can split on the same feature multiple times at different thresholds.
   171|
   172|However, the skill's rules are qualitatively different from decision tree paths. A \textsc{data2skills} rule like ``IF worst\_concave\_points $>$ 0.18 THEN malignant'' is a standalone diagnostic statement that a physician can evaluate independently. A decision tree path through 5 nodes is harder to interpret as a coherent clinical rule.

   192|\subsection{Skill Transfer}
   193|
   194|A skill optimized on Breast Cancer was applied without modification to a held-out test set from a different random split, achieving 86.3\% accuracy—demonstrating that skills, like model weights, capture generalizable patterns rather than memorizing training data.

   196|\section{Related Work}
   197|
   198|\textbf{Interpretable ML.} Decision trees \cite{breiman1984} and rule lists \cite{letham2015} produce interpretable models but scale poorly with feature interactions. SHAP \cite{lundberg2017} and LIME \cite{ribeiro2016} explain black-box predictions post-hoc but do not produce standalone knowledge artifacts.
   199|
   200|\textbf{Text-space optimization.} TextGrad \cite{textgrad2024} introduced automatic differentiation via text for prompt optimization. SkillOpt \cite{skillopt2026} applied text-space optimization to agent skills with rigorous validation gating and learning rate budgets. SkillGrad \cite{skillgrad2026} added momentum-based pattern accumulation for more stable optimization. Our work extends this paradigm from agent skills to data-derived expert knowledge.
   201|
   202|\textbf{Neuro-symbolic methods.} Neural-symbolic integration \cite{garcez2019} combines neural networks with symbolic reasoning, but typically requires predefined symbolic structures. \textsc{data2skills} discovers the symbolic structure (rules) from data through optimization.

   204|\section{Conclusion}
   205|
   206|We presented \textsc{data2skills}, a framework that extracts interpretable expert knowledge from data through gradient-optimized text skills.
   207|By treating a skill document as a differentiable parameter in text space, we enable optimization dynamics analogous to weight-space gradient descent while producing human-readable, auditable, and transferable knowledge artifacts.
   208|
   209|Our experiments demonstrate that text skills achieve competitive accuracy with decision trees on standard benchmarks while producing standalone diagnostic rules that domain experts can independently verify.
   210|The framework supports both LLM-powered and statistical diagnosis, making it deployable across a range of cost and privacy constraints.
   211|
   212|Future work includes LLM-powered diagnosis at scale, multi-skill composition (specialist skills that vote), application to medical imaging and time-series data, and deployment in clinical decision support systems where interpretability is legally mandated.
